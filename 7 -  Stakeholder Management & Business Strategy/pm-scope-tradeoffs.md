# Handling Scope Changes & Trade-Offs (#8)

## Overview

This task focuses on understanding how Product Managers handle scope changes and make trade-offs between user needs, technical feasibility, and timelines.

---

## Goal

To analyse a real example where feature scope changed and evaluate how trade-offs were managed.

---

## Why This Is Important

Scope changes are common during development due to:

- Technical constraints
- New insights
- Time limitations

PMs must balance:

- User value
- Engineering effort
- Delivery timelines

---

## Research & Learn

### 1. Common Reasons for Scope Changes

- Technical complexity higher than expected
- UX design requires simplification
- Time or resource constraints
- New user feedback or insights
- Dependencies between features

---

### 2. Handling Pushback from Engineers

PMs should:

- Understand the technical constraint clearly
- Ask for alternative solutions
- Re-prioritise based on impact vs effort
- Align on a realistic scope

---

### 3. When to Delay vs Reduce Scope

- **Delay feature** → when quality is critical or core functionality is incomplete
- **Reduce scope (MVP)** → when delivering value early is more important

---

## Case Study: Custom Blocking Schedule

---

### What Changed and Why?

#### Original Scope

- Flexible blocking schedules
- Multiple time segments
- Custom block lists
- Integration with routines
- Visual planner integration
- Calendar-based triggers (future)

---

#### Scope Changes

- Visual planner integration → **removed from v1 (out of scope)**
- Calendar-based blocking → **moved to long-term**
- Some advanced flexibility → **simplified for initial release**

---

#### Reason for Changes

- Feature complexity was high
- Risk of overwhelming users
- Need to deliver faster (MVP approach)
- Engineering constraints for backend logic

---

### How Was the Decision Made?

The team prioritised:

- Core functionality (custom schedules + block modes)
- High user value features first
- Simpler implementation for faster release

Trade-off:

- Reduced feature completeness
- Improved speed and feasibility

---

### What Could Have Been Done Differently?

- Define **clear MVP scope earlier**
- Break feature into smaller milestones
- Validate complexity with engineers sooner
- Prototype and test UX earlier to reduce rework

---

## Reflection

### 1. Strategy for Handling Last-Minute Scope Changes

Break the feature into **must-have vs nice-to-have**, and prioritise delivering core value first.

---

### 2. Balancing User Needs vs Technical Constraints

PMs should:

- Focus on solving the **core user problem**
- Accept simpler implementations if they still deliver value
- Avoid over-engineering in early versions

---

### 3. When to Push Back vs Compromise

#### Push Back When:

- Core user value is at risk
- Feature becomes meaningless without key functionality

#### Compromise When:

- Feature can still deliver value in a simpler form
- Technical constraints significantly increase effort or risk

---

## Key Takeaways

- Scope changes are normal and necessary
- MVP thinking helps deliver value faster
- Clear prioritisation reduces confusion and rework
- PMs must balance ambition with feasibility
