# Understanding One-Pagers for Feature Documentation (#11)

## Overview

This task focuses on understanding how one-pagers are used in product management to clearly and efficiently define features.

---

## Goal

To learn how one-pagers help communicate feature requirements with clarity while avoiding unnecessary complexity.

---

## Why This Is Important

One-pagers allow teams to:

- Quickly align on feature goals
- Reduce miscommunication between stakeholders
- Move faster without writing long, complex documents

---

## Research & Learn

### 1. What is a One-Pager in Product Management?

A one-pager is a **short, structured document** that summarises a feature, including:

- Problem
- Solution
- Impact
- Key considerations

It focuses on **clarity and alignment**, not detailed specifications.

---

### 2. Key Sections of a Good One-Pager

Typical sections include:

- **Title** → Feature name
- **Problem** → What user pain point is being solved
- **Why Now** → Why this matters at this time
- **Proposed Solution** → What will be built
- **Hypothesis** → Expected outcome
- **Metrics** → How success is measured
- **Risks** → Potential challenges
- **Next Steps** → What happens next

---

### 3. Use of Visuals

PMs often include:

- Wireframes
- UI mockups (Figma links)
- Flow diagrams

Visuals help:

- Reduce ambiguity
- Improve understanding for developers and designers
- Speed up decision-making

---

### 4. Difference Between One-Pager and PRD

| One-Pager          | PRD                        |
| ------------------ | -------------------------- |
| Short and concise  | Detailed and comprehensive |
| Focus on alignment | Focus on implementation    |
| High-level         | Low-level details included |
| Quick to create    | Time-consuming             |

## Strengths of the One-Pager

- Clear problem statement with user quote
- Strong alignment with user needs (flexibility, control)
- Well-defined solution with practical examples
- Includes hypothesis and measurable metrics
- Provides technical details and backend considerations
- Includes Figma design reference

---

## Areas for Improvement

### 1. Lack of User Flow Explanation

- The document does not clearly explain **how users interact step-by-step**
- A simple flow diagram would improve clarity

---

### 2. Missing Edge Case Considerations

- Scenarios like:
    - Overlapping schedules
    - Conflicts with routines
- These are partially mentioned but not clearly defined

---

### 3. Too Much Technical Detail for a One-Pager

- Backend schema and API details make it closer to a PRD
- One-pager should stay **high-level**

---

### 4. Unclear Scope Boundaries

- Some features are marked “long-term”
- Could be clearer what is included in **v1 vs future**

---

## Suggested Improvements

- Add a **simple user flow diagram**
- Separate **v1 scope vs future features**
- Move backend details to a technical document
- Add **example user scenarios** (e.g., student, worker, weekend use)

## Reflection

### 1. How does a one-pager balance clarity with brevity?

A one-pager balances clarity and brevity by focusing only on **key decisions and outcomes**, avoiding unnecessary technical detail while still providing enough context for alignment.

---

### 2. Most Critical Sections

The most important sections are:

- Problem → defines why the feature matters
- Proposed Solution → defines what will be built
- Metrics → defines success

---

### 3. One Change I Would Make

I would **simplify the technical details and add a user flow diagram**, making the document easier to understand for all stakeholders.

---

## Key Takeaways

- One-pagers are powerful tools for fast alignment
- They should prioritise clarity over completeness
- Visuals significantly improve understanding
- Keeping scope focused is critical
